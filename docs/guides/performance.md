# 性能与资源

本文给出**实测**的耗时基准与内存约束，以及在内存受限的机器上把长任务跑完的调参方法。

> ℹ️ 本文只写有实测记录支撑的数字。没有实测证据的性能声明一律不出现在这里。

## 测量口径

所有基准数据来自同一台开发机，除非另行标注：

| 项 | 值 |
|---|---|
| 平台 | WSL2 Ubuntu |
| 可用内存 | 约 23–24 GB |
| 默认数据窗口 | `20230101–20241231`（约 484 个交易日） |
| 默认票池 | `csi500`（窗口内成分并集 691 支） |
| 计时方式 | wall-clock，重复 2 次取较小值；分段用 `StageTimer` 的 INFO 日志 |

绝对秒数会随机器、缓存冷热、并发负载浮动；**相对量级与瓶颈位置**才是这些数字的用处。

复现方式就是直接跑对应命令并计时：

```bash
timeout 600 /usr/bin/time -f 'WALL_SEC %e' \
  pixi run -- fz factor run momentum_20d --start 20230101 --end 20241231 --universe csi500
```

## 端到端基准

以下为 2026-07-17 一轮全链路性能优化收官时的对账实测。「优化前」是同一台机器上的历史基线，两侧数值结果经等价性验证（挖掘候选表逐字节一致、回测净值 `max|Δ|` 在 1e-15 量级）。

| 场景 | 优化前 | 当前 | 提速 |
|---|---|---|---|
| 单因子评估 e2e（csi500 2 年） | 17.25s | **7.08s** | 2.4× |
| 单因子评估 e2e（全 A 2 年） | 42.13s | **13.84s** | 3.0× |
| 表达式挖掘 50 trials（默认开库正交） | 426.37s | **31.39s** | 13.6× |
| Barra 风险模型构建（csi500 2 年） | 170.98s | **4.14s** | 41× |
| research 链路的风险段（2 个调仓日） | 40.2s | **1.07s** | 37× |
| 日内特征电池：已覆盖月份 | 6s/月 | **0.007s** | 增量跳过 |
| 全 A 回测（隔离测量） | 约 18s | **4.3s** | 4.2× |
| 模拟交易（半年窗、6 次调仓） | — | 1.4s | — |

> ℹ️ 风险模型那一项在提速的同时修掉了一个正确性缺陷：因子集锁定在首个有效截面，导致 484 个交易日里有 451 个被静默丢弃、协方差实际只用 33 天估出。**提速与修 bug 同时发生时，前后数值本就不该相等**——该项的 R² 由 0.287 变为 0.322 是有意的变化。

### 单因子评估的耗时构成

优化前 17.25s 的分布（说明瓶颈在哪，不是当前值）：预处理（MAD → z-score → 中性化 → PIT 过滤）约 8–9s 是最大单段，分位多空回测约 3.9s 次之，universe membership 约 2.3s，IC / 换手 / 报告合计不到 1s。

三个大头分别被消掉：财务数据的 PIT 对齐从按快照日循环改为一次性向量化（6.8s → 0.196s）、行业与市值中性化用矩阵求解替代逐日 OLS（12.92s → 0.34s）、指数成分 membership 改月度缓存 as-of 展开（2.3s → ≤0.5s）。

### 挖掘的耗时构成

优化前 50 trials 要 426s，其中**库正交检查占绝对主导**：对在库的约 84 个因子逐对做相关性检查，10 trials 就要约 220s。改为一次性构建相关面板、全日期批量归约后，同样的 10 trials 降到 17.75s；合成数据上的微基准显示面板路径 0.007s vs 逐对路径 10.404s。

表达式求值本身从来不是瓶颈——单个表达式物化平均 0.015s，50 个批量 0.73s。

### 组件级量级

这些环节在上述优化中未被触及，实测量级仍然成立：

| 环节 | 规模 | 耗时 |
|---|---|---|
| 日频 parquet 加载 | 2.55M × 11 列 | 0.047s |
| 分钟 parquet 加载 | 一个季度、74.9M 行约 4.9 GB | 0.51s（热缓存） |
| 内置因子 `compute`（热上下文） | csi500 2 年 | 0.10–0.21s |
| 单个表达式物化 | 同上面板 | 平均 0.015s |
| IC 分析段 | — | 0.18s |
| 换手率计算 | — | 0.05s |
| Tear Sheet 渲染 | — | 0.67s |
| 组合优化求解 | 单截面 | 0.09s |
| 归因计算 | — | <5ms |

> ✅ **数据加载不是瓶颈。** 日频热缓存下 0.1s 以内，分钟数据按季度读也只要半秒。看到链路慢，先怀疑预处理与库正交，不要先怀疑 I/O。

## 内存才是主约束

在这台 24 GB 的机器上，**耗时早已不是长任务的死因，内存才是**。全 A 长窗口挖掘一度稳定 OOM，而同样的代码在 csi800 上跑得很稳。

原因是规模：全 A 长窗口的基础数据帧本身就有 957 万行，单帧约占 17 GB；挖掘过程中同时存在多份近全宽副本时会直接打穿。经过多轮内存治理（帧列白名单收窄、单副本纪律、`ts_code` 转 Categorical、求值入口按引用叶裁列、相关面板免物化）后，同一配置的稳态占用降到 15 GB 左右并可稳定跑完。

### 已验证可跑的配置

| 配置 | 状态 |
|---|---|
| `csi800` + 2020 年起的长窗口 | 已验证完整闭环，稳定 |
| 全 A + 长窗口 | 需配合下面的子进程隔离；无余量，不建议同机跑别的重任务 |
| 全 A + 长窗口 + 重型分钟嵌套表达式 | 单候选物化尖峰会打穿余量，当前配置下不可行 |

> ⚠️ 提高 WSL 内存上限**不是**通用解法：`.wslconfig` 能分配的上限受宿主机物理内存约束，在 32 GB 宿主机上 24 GB 已接近合理上限。真正的解法是下面的隔离与并发控制。

### 库池的子进程内存隔离

因子库池（把在库因子全部物化成宽面板）是挖掘过程中最大的一次性内存尖峰。把它放进独立子进程，**子进程退出时操作系统全额归还内存**，父进程后续的护栏与评估阶段就有了余量。实测：子进程峰值约 14 GB，退出后可用内存从 6.0 GB 恢复到 17.2 GB；后续从 parquet 缓存装载池只需约 4 秒。

两种用法，效果等价：

```bash
# 用法 A：让 mine team 自己在子进程里建池
pixi run -- fz mine team --start 20200101 --end 20241231 --universe csi800 \
  --iterations 8 --pool-subproc

# 等效环境变量
FACTORZEN_POOL_SUBPROC=1 pixi run -- fz mine team ...
```

```bash
# 用法 B：先离线预构建，多个 session 共享同一份池缓存
pixi run -- fz pool-prebuild --start 20200101 --end 20241231 --universe csi800 \
  --out workspace/factors/_cache/pool_20260718

pixi run -- fz mine team --start 20200101 --end 20241231 --universe csi800 \
  --iterations 8 --pool-subproc
```

> ⚠️ 预构建的**窗口、票池、`--holdout-ratio` 必须与随后的挖掘完全一致**，否则口径不匹配。缓存键包含因子库指纹与窗口指纹，库有变更会自动失效并重建，不会悄悄用旧池。

命令的完整参数见 [CLI 参考 · `fz pool-prebuild`](../reference/cli.md#fz-pool-prebuild)。

### lift 准入的并发

`fz factor-library lift-test` 与团队挖掘收尾的自动 lift 都是逐候选跑 walk-forward 交叉验证，可以并行。并发度直接决定内存占用：**每个 worker 峰值约 3–5 GB**（面板构建的中间物化），再加共享长表 1–2 GB，按 **约 5 GB/worker** 折算。

不指定 `--lift-workers` 时按可用内存自适应：

```text
workers = max(2, min(4, 可用内存GB // 5))
```

上限 **4**；读取可用内存失败时回退 **2**。显式传 `--lift-workers 1` 走纯串行（语义不变，只是慢）。

```bash
# 自适应（推荐）
pixi run -- fz factor-library lift-test --market ashare

# 内存紧张时手动压低
pixi run -- fz factor-library lift-test --market ashare --lift-workers 2
```

> ⚠️ 候选数较多时用 `--lift-workers 1` 会显著拖慢——批量裁决从小时级降到十分钟级正是靠这个并行。命令会在候选数大而并发为 1 时打印告警。

### 日内特征面板构建的并发

`fz data intraday-features build --workers N` 是**月级进程并行**，内存开销比 lift 更陡：**单月峰值约 7.6 GiB**。

- 24 GiB 内存的机器建议最多设 `2`。
- 设置 `>2` 会打印告警。

已覆盖的月份会被增量跳过（6s/月 → 0.007s），所以补跑历史时真正付出并行代价的只有缺口月份。

## 并发旋钮一览

| 旋钮 | 所属命令 | 默认 | 并发类型 | 内存代价 |
|---|---|---|---|---|
| `--workers` | `fz mine search` | `1` | 遗传搜索评分线程 | 低；同 seed 下与串行结果等价 |
| `--workers` | `fz data intraday-features build` | `1` | 月级进程 | **高：约 7.6 GiB/月** |
| `--llm-workers` | `fz mine team` | `4` | 轮内 LLM 调用 | 可忽略（网络等待） |
| `--lift-workers` | `fz mine team` · `fz factor-library lift-test` | 自适应 | 候选级线程 | **约 5 GB/worker，上限 4** |
| `--pool-subproc` | `fz mine team` | 关 | 池构建放子进程 | **降低**父进程峰值 |

> ✅ 判断原则：**LLM 并发几乎免费，进程/线程级计算并发按内存买单。** 内存紧张时优先关掉 `intraday-features build --workers` 与压低 `--lift-workers`，而不是压 `--llm-workers`。

## 调优检查清单

按顺序自查，命中前面的就不用看后面的：

1. **任务被 OOM kill 了？** 先开 `--pool-subproc`（或先 `fz pool-prebuild`），再压 `--lift-workers`，最后缩票池到 `csi800`。
2. **挖掘慢得离谱？** 确认是否开着库正交检查。scout 阶段可以 `--no-library-orthogonal` 快速探索，正式入库前再开回来。
3. **单因子评估慢？** 看日志的分段计时，绝大多数情况瓶颈在预处理段而非 I/O。
4. **补历史日内特征慢？** 确认增量跳过生效（已覆盖月应是毫秒级）；只有缺口月份才需要 `--workers`。
5. **回测慢？** 只要净值不要持仓明细时，关掉持仓/成交明细的收集可以走更省的路径。

## 已裁决为「不值得优化」的项

这些经过实测后判定收益不成立，记录在此以免重复投入：

| 项 | 裁决 | 依据 |
|---|---|---|
| **GPU 加速** | 暂缓 | 热点是算法结构（Python 循环、重复计算），不是算子吞吐。日频 csi500 规模（25–45 万行）喂不饱 GPU，PCIe 传输开销大于收益；分钟全量数据也远超单卡显存。 |
| **列式数据库迁移** | 暂缓 | 审计结论是「定向双轨」：按标的的时序查询快 2.5–9×，但截面扫描 parquet 反而快 8×，分钟级数据不值得迁。收益不覆盖迁移与双写成本。 |
| 组合优化求解 | 无需优化 | 单截面 0.09s。 |
| 归因计算 | 无需优化 | <5ms。 |
| crypto 链路 | 无专属热点 | 与 A 股共用同一批已优化的引擎。 |
